当前项目关系
    Socket：封装 fd，析构函数会 close(fd_)。所以它是“拥有 fd”的类。

    InetAddress：封装 sockaddr_in，负责 IP/端口转换。

    Epoll：封装 epollfd_，负责 epoll_ctl 和 epoll_wait。

    Channel：封装某一个 fd 在 epoll 上的状态。它本身不拥有 fd，也不拥有 Epoll，只保存：
            fd_          // 哪个 fd
            ep_          // 属于哪个 Epoll
            events_      // 想监听哪些事件
            revents_     // epoll_wait 实际返回了哪些事件
            isInEpoll_   // 是否已经添加到 epoll 树
    Conn：保存客户端连接的业务状态，比如输出缓冲区、是否半关闭、对应的 Channel*。

Channel 加进来之后，项目应该这样理解

  以前是直接把 fd 放进 epoll：

  ev.data.fd = fd;

  所以 epoll_wait 返回后，只能拿到 fd。

  现在改成了：

  ev.data.ptr = chn;

  也就是说，epoll 里保存的不再只是 fd，而是这个 fd 对应的 Channel*。这样 epoll_wait 返回后，你能拿到完整的 Channel：

  Channel* chn = (Channel*)events_[i].data.ptr;

  Channel 里有：

  fd_          // 这个 Channel 对应哪个 fd
  ep_          // 属于哪个 Epoll
  events_      // 想监听什么事件，比如 EPOLLIN / EPOLLOUT
  revents_     // 实际发生了什么事件
  isInEpoll_   // 是否已经加入 epoll

  核心流程是：

  1. main 创建监听 socket。
  2. main 创建 Epoll ep;。
  3. main 创建监听 fd 对应的 Channel serverChn。
  4. serverChn->set__events_(EPOLLIN);
  5. serverChn->update();
  6. Channel::update() 调用 Epoll::UpdateChannel(this)。
  7. Epoll::UpdateChannel() 把 Channel* 塞进 epoll：

  ev.data.ptr = chn;
  ev.events = chn->get__events_();

  8. 后面 ep.loop(-1) 调用 epoll_wait。
  9. Epoll::loop() 从 epoll 返回值里取出 Channel*：

  Channel* chn = (Channel*)this->events_[i].data.ptr;
  chn->set__revents_(this->events_[i].events);
  chns.push_back(chn);

  10. main 遍历这些 Channel*，判断：

  if(chns[i]->getfd_() == sockServer.getfd())

  如果是监听 fd，说明有新连接；否则就是客户端 fd 有读写/关闭/错误事件。

  为什么加 Channel 后更绕

  因为你从“epoll 管 fd”变成了“epoll 管 Channel”。

  旧模型：

  fd -> epoll -> epoll_wait -> fd

  新模型：

  fd -> Channel -> epoll -> epoll_wait -> Channel -> fd / events / revents

  这样做的好处是，以后你可以继续给 Channel 加回调函数，比如：

  readCallback_
  writeCallback_
  closeCallback_
  errorCallback_

  那主循环就不用自己判断这么多 if(events & EPOLLIN)，可以直接：

  chn->handleEvent();

  这就是视频教程后面通常会讲的 Reactor 模型。



一个 epollfd 可以管理多个 fd
一个 fd 通常对应一个 Channel
一个 Channel 只对应一个 fd
一个 Channel 属于一个 Epoll
一个 Epoll 对象内部有一个 epollfd

  画出来：

  Epoll ep

  epollfd_
    |

  Epoll对象 1 个

  fd 多个

  服务端启动流程：

  1. tcpepoll.cpp:312 main() 检查参数。
  2. Socket.cpp:3 createNonBlockingFd() 创建非阻塞 listenfd。
  3. tcpepoll.cpp:329 Socket sockServer(listenfd) 用 RAII 管理监听 fd。
  4. 设置 SO_REUSEADDR、SO_REUSEPORT、SO_KEEPALIVE。
  5. InetAddress.cpp:9 InetAddress server(ip, port) 生成 sockaddr_in。
  6. sockServer.Bind(server) 调 bind()。
  7. sockServer.Listen(128) 调 listen()。
  8. Epoll.cpp:13 Epoll ep 创建 epollfd_。
  9. tcpepoll.cpp:420 创建 serverChn，它代表监听 fd。
  10. serverChn->setReadCallback(...) 把监听 fd 的读事件绑定到 handleAccept()。
  11. serverChn->set__events_(EPOLLIN) 表示监听新连接。
  12. serverChn->update() 调到 Epoll.cpp:36，最终 epoll_ctl(ADD)。

  事件循环流程：

  1. tcpepoll.cpp:449 ep.loop(-1) 调用 epoll_wait()。
  4. main() 遍历 Channel*，调用 chns[i]->handleEvent()。
  5. Channel::handleEvent() 根据 revents_ 调用对应回调。

  新连接流程：

  1. 监听 fd 发生 EPOLLIN。
  2. serverChn->handleEvent() 调 readCallback_。
  3. readCallback_ 实际执行 handleAccept(&ep, &sockServer, epollfd, opt)。
  4. handleAccept() 循环 accept4()，直到 EAGAIN。
  5. 每个 clientfd 创建一个 Conn。
  6. 每个 clientfd 创建一个 Channel。
  7. 给客户端 Channel 绑定读、写、关闭、错误回调。
  8. 把客户端 Channel 加入 epoll，监听 EPOLLIN | EPOLLET | EPOLLRDHUP。

  读写流程：

  1. 客户端 fd 有 EPOLLIN。
  2. handleClientRead() 循环 read()，直到 EAGAIN。
  3. 读到的数据追加到 conns[fd].outbuf。
  4. tryWrite() 尝试发送。
  5. 如果一次发不完，refreshConn() 给这个 fd 加上 EPOLLOUT。
  6. 下次 fd 可写时，handleClientWrite() 继续发送 outbuf。
  7. outbuf 发空后，取消 EPOLLOUT，避免 CPU 空转。

  关闭流程：

  1. read() 返回 0，或者收到 EPOLLRDHUP/EPOLLHUP。
  2. 设置 peerClosed = true。
  3. 如果 outbuf 还有数据，先继续发。
  4. 如果 peerClosed == true && outbuf.empty()，调用 closeConn()。
  5. closeConn() 先 epoll_ctl(DEL)，再 close(fd)，再 delete Channel，最后从 conns 删除连接状态。

  各类职责
  InetAddress：封装 sockaddr_in。负责 IP 字符串、网络字节序端口、本地 sockaddr* 之间转换。它不拥有 fd。

  Socket：封装 socket fd。析构函数会 close(fd_)，所以它是 RAII 所有者。监听 socket 适合用 Socket 管；客户端 fd 当前由 conns + closeConn() 管，暂时不要再包成临时 Socket*。

  Epoll：封装 epollfd_。构造时 epoll_create1()，析构时 close(epollfd_)。UpdateChannel() 根据 Channel::isInEpoll_ 决定 ADD 还是 MOD。loop() 返回活跃的 Channel*。

  Channel：一个 fd 对应一个 Channel。它不拥有 fd，也不拥有 Epoll，只记录 fd_、events_、revents_、isInEpoll_ 和四类回调。它的作用是把“fd + 事件 + 处理函数”绑在一起。

  Conn：业务连接状态。保存 outbuf、peerClosed、channel。它解决 TCP 非阻塞写“不一定一次发完”的问题。

  关键 C++ 语法
  RAII：Socket::~Socket() 和 Epoll::~Epoll() 是 RAII。对象生命周期结束时自动释放系统资源。好处是少忘记 close()，坏处是你必须清楚谁拥有 fd。

  = delete：Socket(const Socket&) = delete; 防止复制。因为复制一个拥有 fd 的对象会导致两个对象析构时关闭同一个 fd。

  std::function<bool()>：Channel 用它保存回调函数。bool 表示事件处理后连接/Channel 是否还有效。

  lambda 回调：例如：

  serverChn->setReadCallback([&ep, &sockServer, epollfd, opt](){
      return handleAccept(&ep, &sockServer, epollfd, opt);
  });

  [&ep, &sockServer, epollfd, opt] 是捕获列表。ep 和 sockServer 按引用捕获，epollfd 和 opt 按值捕获。之后 Channel::handleEvent() 不需要知道业务细节，只需要调用 readCallback_()。

  epoll_event.data.ptr：以前你用 data.fd，epoll 返回后只能拿到 fd。现在用 data.ptr = chn，epoll 返回后直接拿到完整 Channel*，这就是 Reactor 模型的雏形。

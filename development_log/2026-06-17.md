## 今日完成

### 阶段 1B：Channel 持有 Eventloop*

  1. `Channel` 构造函数参数从 `Epoll*` 改为 `Eventloop*`。
  2. `Channel` 内部成员从 `Epoll*` 改为 `Eventloop*`。
  3. `Channel::update()` 通过 `Eventloop::updateChannel()` 转发。
  4. `Channel::remove()` 通过 `Eventloop::removeChannel()` 转发。
  5. `TcpServer` 创建 `listenChannel_` 时传入 `Eventloop*`。
  6. `TcpConnection` 创建 `channel_` 时传入 `Eventloop*`。

### 阶段 1C：Eventloop 接管事件循环

  1. `Eventloop` 新增 `loop()` 函数。
  2. `Eventloop::loop()` 内部调用 `poll(-1)` 等待活跃事件。
  3. `Eventloop::loop()` 遍历活跃的 `Channel*`。
  4. `Eventloop::loop()` 调用 `Channel::handleEvent()` 分发事件。
  5. `Eventloop` 通过删除连接回调通知 `TcpServer` 删除失效连接。
  6. `TcpServer::start()` 改为调用 `Eventloop::loop()`。
  7. `TcpServer` 仍然负责 `conns_` 连接集合和 `TcpConnection` 生命周期。

## 遇到的问题

1. 阶段 1A 后，Eventloop 已经拥有 Epoll，但 Channel 仍然直接依赖 Epoll*。
2. 阶段 1B 后，事件循环主体仍然写在 TcpServer 中，Eventloop 还没有真正负责事件分发。
3. Eventloop 接管事件分发后，连接删除不能直接由 Eventloop 完成

## 问题原因

1. Channel 负责 fd 的事件状态，Epoll 负责底层 epoll_ctl/epoll_wait，中间应该由 Eventloop 统一协调。
2. TcpServer 应该负责连接管理，不应该继续负责主事件循环和事件分发。

## 解决思路

1. 让 Channel 只依赖 Eventloop，由 Eventloop 转发 updateChannel/removeChannel 到内部 Epoll。
2. 将 poll(-1)、活跃 Channel* 遍历、Channel::handleEvent() 调用移动到 Eventloop::loop() 中。
3. 连接删除仍由 TcpServer 负责，Eventloop 只通过回调通知。

当前架构中：

```text
  main
    -> Eventloop
         -> Epoll
         -> loop()
              -> poll(-1)
              -> Channel::handleEvent()
              -> removeConnectionCallback(fd)
    -> TcpServer
         -> listen Socket
         -> listen Channel
         -> unordered_map<int, unique_ptr<TcpConnection>>
                -> TcpConnection
                     -> client Socket
                     -> client Channel
```

## 今日学到的知识点

1. Eventloop 是 Reactor 的事件循环核心。
2. Channel 不应该直接管理 Epoll。
3. TcpServer 应该负责连接管理，不应该负责事件分发。
4. std::function 可以保存跨对象回调。
5. lambda 捕获 [this] 可以让回调调用当前对象成员函数。
6. TcpConnection 的生命周期仍然由 TcpServer::conns_ 管理。
7. unique_ptr 管理对象生命周期，conns_.erase(fd) 会触发 TcpConnection 析构

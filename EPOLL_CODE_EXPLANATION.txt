tcpepoll 代码修改说明和完整讲解
==================================

一、这次主要改了什么
--------------------

这次改了两个源码文件：

1. src/tcpepoll.cpp
   这是服务端，主要修复 epoll + 非阻塞 socket 下的写数据问题。

2. src/client.cpp
   这是客户端，主要修复 TCP 字节流下 send/recv 不保证一次完成的问题。

这次没有推翻你原来的代码结构。你原本写对的地方，比如：

- socket()
- bind()
- listen()
- epoll_create1()
- epoll_ctl(ADD)
- epoll_wait()
- accept()
- 客户端 connect()
- 服务端 read() 循环读到 EAGAIN
- 设置非阻塞
- 设置 TCP_NODELAY
- 使用 EPOLLET
- 使用 MSG_NOSIGNAL

这些都尽量保留了。重点是把“不完整的非阻塞写流程”补完整。


二、原来的核心问题是什么
------------------------

原来最大的问题不是编译问题，而是 TCP 和非阻塞 I/O 的语义没有处理完整。

TCP 不是消息队列，而是字节流。

比如客户端执行一次：

    send(sockfd, "hello", 5, ...)

服务端不一定一次 read() 就刚好读到 5 字节。它可能读到：

    "he"

下一次再读到：

    "llo"

反过来也一样。服务端执行一次 send()，客户端也不一定一次 recv() 就收到完整数据。

另一个重点是：非阻塞 socket 的 send() 不保证把你传入的数据全部发完。

例如服务端想 echo 回 1024 字节：

    send(fd, buf, 1024, MSG_NOSIGNAL)

可能只成功发了 300 字节，然后返回 300。

这不是错误。它只表示当前内核发送缓冲区暂时装不下更多数据了。

如果再继续 send()，可能得到：

    -1, errno = EAGAIN

EAGAIN 的意思不是连接坏了，而是：

    现在不能继续写，等以后 fd 可写了再写。

所以 epoll 非阻塞写的正确流程是：

    要发送的数据先放进应用层输出缓冲区 outbuf
    尝试 send()
    如果没发完，就保留剩余数据
    给这个 fd 监听 EPOLLOUT
    以后 epoll 通知 EPOLLOUT 时，再继续发送 outbuf
    outbuf 发空后，取消 EPOLLOUT

你之前已经定义了 outbuf 和 tryWrite()，但主循环没有完全按这个模型走。


三、服务端 src/tcpepoll.cpp 改动说明
------------------------------------

1. Conn 结构体增加 peerClosed

现在是：

    struct Conn {
        std::string outbuf;
        bool peerClosed = false;
    };

outbuf 用来保存“还没发完的数据”。

peerClosed 用来记录“对端已经关闭写端”。

为什么需要 peerClosed？

因为 EPOLLRDHUP 或 EPOLLHUP 到来时，可能还有数据没读完，或者还有 echo 数据没发完。

如果一看到 EPOLLRDHUP 就立刻 close(fd)，可能会丢最后一批数据。

现在的策略是：

    对端关闭了 -> 标记 peerClosed = true
    先把已经读到的数据 echo 回去
    outbuf 为空后再关闭连接


2. 删除服务端原来的 sendAll()

原来的 sendAll() 看起来像是“发送全部数据”，但在非阻塞 socket 下它遇到 EAGAIN 会提前返回。

提前返回本身没错，但调用者必须保存没发完的数据。

服务端现在统一使用：

    conns[fd].outbuf
    tryWrite()
    EPOLLOUT

所以原来的 sendAll() 不再适合服务端主流程。


3. tryWrite() 的作用

tryWrite() 只做一件事：

    尽量把 conns[fd].outbuf 里的数据发出去。

它的循环逻辑是：

    while outbuf 不为空:
        send()
        如果 send 成功发了 n 字节:
            从 outbuf 头部删除 n 字节
        如果 errno == EINTR:
            被信号打断，继续重试
        如果 errno == EAGAIN 或 EWOULDBLOCK:
            当前不能继续写，但连接没坏，返回 true
        其他错误:
            关闭连接，返回 false

注意：

    EAGAIN 不是失败。

它表示：

    这次先写到这里，剩下的以后等 EPOLLOUT 再写。


4. modEvent() 的作用

modEvent() 用来修改某个客户端 fd 在 epoll 中关心的事件。

什么时候需要 EPOLLOUT？

只有 outbuf 不为空时才需要。

也就是：

    outbuf 有数据没发完 -> 监听 EPOLLOUT
    outbuf 已经发空 -> 不监听 EPOLLOUT

为什么不能一直监听 EPOLLOUT？

因为 TCP socket 大多数时候都是可写的。如果一直监听 EPOLLOUT，epoll_wait() 会频繁返回，导致 CPU 空转。


5. listenfd 设置非阻塞后检查返回值

原来调用：

    setnonblocking(listenfd);

但没有判断是否失败。

现在改成：

    if(setnonblocking(listenfd) == -1) {
        perror("setnonblocking listenfd");
        close(listenfd);
        return -1;
    }

这样如果 fcntl() 失败，程序不会继续带着错误状态运行。


6. accept() 改成循环 accept 到 EAGAIN

现在 listenfd 是非阻塞的。

当 epoll 告诉你 listenfd 可读时，含义是：

    监听队列里至少有一个新连接。

但可能不止一个。

所以正确写法是：

    while true:
        accept()
        成功就处理这个 clientfd
        EINTR 就继续
        EAGAIN/EWOULDBLOCK 表示连接已经 accept 完了，break
        其他错误才是真的 accept 失败

这也是 epoll 非阻塞服务器常见写法。


7. 每个新连接都放进 conns

新客户端接入后，现在会执行：

    conns[clientfd] = Conn{};

这样这个 fd 就有自己的 outbuf 和 peerClosed 状态。

服务器必须为每个连接单独保存状态，因为每个客户端的发送进度都可能不同。


8. 客户端事件处理顺序调整

现在客户端 fd 的事件处理顺序是：

    EPOLLERR
    EPOLLIN / EPOLLPRI
    EPOLLRDHUP / EPOLLHUP
    EPOLLOUT
    根据 outbuf 和 peerClosed 决定是否关闭或修改事件

重点是：

    不再一看到 EPOLLRDHUP 就立即 close。

因为 EPOLLRDHUP 可能和 EPOLLIN 同时出现。

例如：

    events = EPOLLIN | EPOLLRDHUP

这表示：

    对方已经关闭写端，但内核缓冲区里还有数据可以读。

所以要先 read()，再决定是否关闭。


9. 读事件中的逻辑

读到数据后，现在不直接 send()，而是：

    conns[fd].outbuf.append(buf, n);
    needWrite = true;

意思是：

    这些数据是要 echo 回客户端的，先进入输出缓冲区。

读循环结束后，如果 needWrite 为 true，就调用：

    tryWrite(epollfd, fd)

如果一次发不完，剩余数据还留在 outbuf 里。

后面会把 EPOLLOUT 加入 epoll 监听，等下次可写时继续发。


10. 写事件中的逻辑

如果 events 里有 EPOLLOUT，说明：

    这个 socket 现在可写，可以继续发送 outbuf。

于是调用：

    tryWrite(epollfd, fd)

如果 outbuf 发空了，就不再监听 EPOLLOUT。

如果 outbuf 还有数据，就继续监听 EPOLLOUT。


四、客户端 src/client.cpp 改动说明
----------------------------------

1. 增加 sendAll()

客户端原来是：

    send(sockfd, buf, strlen(buf), MSG_NOSIGNAL)

问题是 send() 可能只发一部分。

现在改成 sendAll()：

    while total < len:
        send()
        成功发 n 字节 -> total += n
        EINTR -> 重试
        其他错误 -> 返回 false

这样可以保证输入的一整段数据都发出去。


2. 增加 recvAll()

客户端原来是：

    recv(sockfd, buf, sizeof(buf), 0)

问题是 recv() 不保证一次收到完整 echo。

现在客户端知道自己刚才发了 len 字节，所以就循环接收 len 字节：

    while total < len:
        recv()
        成功收 n 字节 -> total += n
        n == 0 -> 服务端关闭连接
        EINTR -> 重试
        其他错误 -> 返回 -1

这和 echo server 的协议匹配：

    发多少字节，就期待收回多少字节。


3. recvbuf 和 send buf 分开

现在发送用 buf，接收用 recvbuf。

这样逻辑更清楚：

    buf 是用户输入
    recvbuf 是服务端返回内容


五、现在服务端完整执行流程
--------------------------

1. 创建监听 socket

    listenfd = socket(AF_INET, SOCK_STREAM, 0)

2. 设置 socket 选项

    SO_REUSEADDR
    SO_REUSEPORT
    SO_KEEPALIVE

3. 设置 listenfd 为非阻塞

    setnonblocking(listenfd)

4. bind()

    绑定 IP 和端口。

5. listen()

    开始监听新连接。

6. epoll_create1()

    创建 epoll 实例。

7. 把 listenfd 加入 epoll

    listenfd 使用 EPOLLIN，表示有新连接到来时通知。

8. 进入事件循环

    epoll_wait() 阻塞等待事件。

9. 如果是 listenfd 事件

    循环 accept()，直到 EAGAIN。

    每 accept 一个 clientfd：

        设置非阻塞
        设置 TCP_NODELAY
        创建 Conn 状态
        把 clientfd 加入 epoll

10. 如果是客户端 fd 事件

    先看 EPOLLERR。

    如果有 EPOLLIN：

        循环 read()
        读到数据就 append 到 outbuf
        读到 EAGAIN 表示这次读完

    如果有 EPOLLRDHUP 或 EPOLLHUP：

        标记 peerClosed = true

    如果需要写，或者有 EPOLLOUT：

        tryWrite()

    最后：

        如果 peerClosed == true 且 outbuf 为空，关闭连接
        如果 outbuf 不为空，监听 EPOLLOUT
        如果 outbuf 为空，取消 EPOLLOUT


六、几个关键概念
----------------

1. 非阻塞 read()

非阻塞 read() 可能返回：

    n > 0
        读到了 n 字节。

    n == 0
        对端关闭了连接，或者关闭了写端。

    n == -1 且 errno == EINTR
        被信号打断，继续重试。

    n == -1 且 errno == EAGAIN/EWOULDBLOCK
        当前已经没有数据可读，不是错误。

    其他 n == -1
        真正的错误，需要关闭连接。


2. 非阻塞 send()

非阻塞 send() 可能返回：

    n > 0
        成功发送了 n 字节，但不一定是全部。

    n == -1 且 errno == EINTR
        被信号打断，继续重试。

    n == -1 且 errno == EAGAIN/EWOULDBLOCK
        当前写缓冲区满了，稍后等 EPOLLOUT。

    其他 n == -1
        真正的错误，需要关闭连接。


3. EPOLLET

EPOLLET 是边缘触发。

边缘触发的意思是：

    状态变化时通知你一次。

所以读事件来了之后，必须一直 read()，直到 EAGAIN。

你原来的读循环这一点是对的，这次保留了。


4. EPOLLOUT

EPOLLOUT 表示 fd 当前可写。

但不要一直监听 EPOLLOUT。

正确做法是：

    有待发送数据时才监听 EPOLLOUT。
    没有待发送数据时取消 EPOLLOUT。


5. EPOLLRDHUP

EPOLLRDHUP 表示对端关闭了写端。

但它不代表“现在立刻不能读了”。

它可能和 EPOLLIN 一起出现，所以应该先读完内核缓冲区里的数据，再考虑关闭。


七、现在代码还可以继续优化的地方
--------------------------------

1. outbuf.erase(0, n) 在大量数据时性能一般

std::string 从头部 erase 会移动后面的数据。

学习阶段没问题。

以后可以用：

    std::vector<char> + 写偏移
    std::deque<char>
    ring buffer

2. 协议还很简单

现在客户端用“发多少字节就收多少字节”的方式测试 echo。

真实业务协议通常会设计：

    固定长度头部 + body 长度
    分隔符协议
    HTTP 协议

3. setsockopt() 的返回值还可以检查

现在为了保持代码简单，没有把每个 setsockopt() 都改成严格检查。

工程级代码里建议检查返回值。

4. 日志可以更规范

现在还是 printf/perror。

后续可以封装日志函数，带时间、fd、错误码等信息。


八、一句话总结
--------------

这次最重要的修改是：

    服务端不再假设 send() 一次能发完。

而是改成：

    数据先进 outbuf
    能发多少发多少
    发不完就监听 EPOLLOUT
    下次可写时继续发
    发完后取消 EPOLLOUT

这就是 epoll 非阻塞写的核心。
